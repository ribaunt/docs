Sobald ein Browser eine Ribaunt-Challenge gelöst hat und Ihr Server die Lösung akzeptiert hat, sollte dieses Token nie wieder akzeptiert werden. Ohne Replay-Schutz könnte ein Angreifer, der ein gültiges `{ tokens, solutions }`-Paar abfängt oder aufzeichnet, es beliebig oft erneut übermitteln, bis das Token abläuft – und damit die gesamte Proof-of-Work-Anforderung umgehen. Ribaunt begegnet dem, indem es verbrauchte Token-IDs (`jti`-Werte) nachverfolgt und jedes bereits gesehene Token ablehnt.

## Replay-Verhinderungs-Modi

Ribaunt unterstützt drei Modi für die Replay-Verhinderung, konfiguriert über die Option `replayPrevention` von `verifySolution()`.

| Modus | Verhalten | Wann verwenden |
| --- | --- | --- |
| `local` (Standard) | Verfolgt verwendete Tokens in einer In-Memory-`Map` innerhalb des aktuellen Node.js-Prozesses | Single-Process-Deployments |
| `remote` | Delegiert an Ihren eigenen verteilten Store über das `ReplayStore`-Interface | Serverless-Funktionen, horizontal skalierte Dienste, mehrere Node.js-Instanzen |
| `disabled` | Überspringt Replay-Prüfungen vollständig | Nur, wenn eine andere Schicht bereits Replay verhindert, oder als Legacy-Opt-out während einer Migration |

## Lokaler Modus (Standard)

Der lokale Modus erfordert keine Konfiguration. Wenn Sie `verifySolution()` ohne die Option `replayPrevention` aufrufen, verwendet Ribaunt automatisch eine prozesslokale `Map`, um verbrauchte `jti`-Werte zu speichern. Jeder Versuch, dasselbe Token zweimal innerhalb desselben Prozesses zu übermitteln, wird abgelehnt.

```ts
// Default behavior — local replay protection is active
const valid = await verifySolution(tokens, solutions);
```

Dies ist die richtige Wahl für die meisten traditionellen Node.js-Deployments, bei denen ein einzelner langlaufender Prozess alle Anfragen bearbeitet. Abgelaufene Token-IDs werden automatisch bereinigt, sobald neue Verifizierungen stattfinden, sodass die Speichernutzung begrenzt bleibt.

<Tip>
  Sie müssen nichts Zusätzliches konfigurieren, um vom lokalen Replay-Schutz zu profitieren – er ist standardmäßig aktiviert.
</Tip>

## Remote-Modus für verteilte Deployments

Wenn Ihre Anwendung über mehrere Node.js-Instanzen läuft, Serverless-Funktionen nutzt oder auf Edge Workern deployt wird, hat jeder Prozess seinen eigenen Speicher. Der lokale Modus kann keinen Zustand zwischen ihnen teilen, was bedeutet, dass ein von einer Instanz verbrauchtes Token gegen jede andere Instanz wiedergespielt werden kann. In diesen Fällen benötigen Sie den Modus `remote`.

Mit dem Modus `remote` stellen Sie ein `replayStore`-Objekt bereit, das das `ReplayStore`-Interface implementiert. Ribaunt ruft für jedes verifizierte Token `replayStore.consume(jti, expiresAt)` auf. Ihre Implementierung ist verantwortlich für die atomare „Als verwendet markieren"-Operation – das Standardmuster mit Redis oder Valkey ist `SET NX EX` (setzen, wenn nicht vorhanden, mit Ablauf).

```ts
const valid = await verifySolution(tokens, solutions, {
  replayPrevention: 'remote',
  replayStore: {
    consume: async (jti, expiresAt) => {
      // Returns true if consumed (first time), false if already seen.
      // Implement with Redis/Valkey SET NX EX for atomic "set if not exists".
      return true;
    },
  },
});
```

Der Wert `expiresAt` ist ein Unix-Zeitstempel in Sekunden – übergeben Sie ihn direkt als TTL für Ihren Cache-Schlüssel, damit Einträge automatisch ablaufen und Sie keine veralteten Daten ansammeln.

<Tip>
  Verwenden Sie das Argument `expiresAt`, um die TTL für Ihren Redis/Valkey-Schlüssel zu setzen. So bleibt Ihr Store ohne manuelle Aufräumarbeiten sauber.
</Tip>

## Replay-Schutz deaktivieren

<Warning>
  Deaktivieren Sie den Replay-Schutz nur, wenn ein separater Mechanismus in Ihrem Stack die Wiederverwendung von Tokens bereits verhindert. Mit deaktiviertem Replay-Schutz kann eine gültige Lösung mehrfach innerhalb des TTL-Fensters des Tokens übermittelt werden.
</Warning>

Wenn Sie sich zuvor auf das alte Verhalten verlassen haben, bei dem `verifySolution()` keine Replay-Prüfungen hatte, können Sie dieses Verhalten während Ihrer Migration explizit wiederherstellen:

```ts
const valid = await verifySolution(tokens, solutions, {
  replayPrevention: 'disabled',
});
```

## ReplayStore-Interface

Bei Verwendung des Modus `remote` muss Ihr `replayStore` das folgende Interface erfüllen:

```ts
interface ReplayStore {
  consume(jti: string, expiresAt: number): Promise<boolean>;
}
```

| Parameter | Typ | Beschreibung |
| --- | --- | --- |
| `jti` | `string` | Die eindeutige ID des zu verifizierenden Challenge-Tokens |
| `expiresAt` | `number` | Unix-Zeitstempel in Sekunden, wann dieses Token abläuft |

Geben Sie `true` zurück, um die Verifizierung fortzusetzen (dies ist das erste Mal, dass dieses Token gesehen wurde). Geben Sie `false` zurück, um es als Replay abzulehnen. Die Operation muss atomar sein – verwenden Sie einen einzelnen `SET NX EX`-Redis-Befehl anstelle einer separaten Check-then-Set-Sequenz, um Race Conditions unter gleichzeitiger Last zu vermeiden.
