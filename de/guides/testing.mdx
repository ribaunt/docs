Ribaunt bietet `solveChallenge()` — einen synchronen, serverseitigen Solver —, der es einfach macht, automatisierte Tests für Ihre Challenge- und Verify-Endpoints ohne Browser zu schreiben. Anstatt einen Headless-Browser einzurichten oder interne Zustände zu mocken, können Sie den vollständigen Zyklus Challenge → Solve → Verify direkt aus Ihrer Test-Suite steuern.

## solveChallenge in Tests verwenden

`solveChallenge` führt denselben Proof-of-Work-Algorithmus aus, den der Browser ausführt, jedoch synchron und in Node.js. Sie können eine Challenge auf Ihrem Server generieren, sie wie ein Client lösen und das Ergebnis dann an `verifySolution` weitergeben — alles innerhalb eines einzigen Tests.

```ts
import { createChallenge, solveChallenge, verifySolution } from 'ribaunt';

// Generate a challenge (as your server would)
const tokens = createChallenge(3, 2, 30); // difficulty 3, 2 challenges, 30s TTL

// Solve it (as the browser would)
const solutions = solveChallenge(tokens);

// Verify the solution (as your verify endpoint would)
const valid = await verifySolution(tokens, solutions);
console.assert(valid === true);
```

<Tip>
  Verwenden Sie in Tests eine niedrige Schwierigkeit (3–4), um sie schnell zu halten. Eine höhere Schwierigkeit verlangsamt CPU-gebundene Tests. Schwierigkeit 5 ist für die Produktion in Ordnung, verlangsamt aber Testsuiten spürbar.
</Tip>

## Testen mit Timeout-Schutzmechanismen

Wenn Sie ein Sicherheitsnetz gegen einen ausufernden Solver in CI benötigen, übergeben Sie Guardrail-Optionen an `solveChallenge`. Sie können die Wanduhrzeit, die Anzahl der Hash-Versuche oder beides begrenzen.

```ts
const solution = solveChallenge(token, {
  maxDurationMs: 2000,    // Give up after 2 seconds
  maxIterations: 500_000, // Or after 500k attempts
});
```

Wenn eines der Guardrails erreicht wird, gibt `solveChallenge` für dieses Token `undefined` zurück. Ihr Test sollte sicherstellen, dass der Rückgabewert definiert ist, bevor Sie ihn an `verifySolution` weitergeben.

## Replay-Schutz testen

Ribaunts Standard-`local`-Replay-Modus blockiert die mehrfache Verifizierung eines Tokens innerhalb desselben Prozesses. Sie können dieses Verhalten direkt in Ihren Tests bestätigen, indem Sie dieselbe Lösung zweimal einreichen.

```ts
const tokens = createChallenge(3, 2, 30);
const solutions = solveChallenge(tokens);

const first = await verifySolution(tokens, solutions);
console.assert(first === true);  // Passes

const second = await verifySolution(tokens, solutions);
console.assert(second === false); // Rejected — replay detected
```

Der zweite Aufruf gibt `false` zurück, weil die Challenge-JTIs beim ersten Gebrauch verbraucht werden.

## Testen mit deaktiviertem Replay-Schutz

Einige Unit-Tests müssen dieselben Token über mehrere Assertions hinweg wiederverwenden — zum Beispiel, um zu testen, dass eine gültige Lösung Ihre Geschäftslogik unabhängig vom Replay-Zustand immer besteht. Sie können den Replay-Schutz für einen einzelnen `verifySolution`-Aufruf deaktivieren.

```ts
const valid = await verifySolution(tokens, solutions, {
  replayPrevention: 'disabled',
});
```

<Note>
  Verwenden Sie `replayPrevention: 'disabled'` nur in Tests. Deaktivieren Sie niemals den Replay-Schutz in Ihrem Produktions-Verify-Endpoint, da dies Angreifern ermöglicht, abgefangene Lösungen wiederzuverwenden.
</Note>

## Verifizierungswarnungen erfassen

Wenn Sie behaupten möchten, _warum_ eine Verifizierung fehlgeschlagen ist — nicht nur, dass sie `false` zurückgegeben hat —, verwenden Sie den `onWarning`-Callback. Dies ist besonders nützlich, um Edge Cases wie abgelaufene Token oder ungültige Lösungen zu testen.

```ts
const warnings: string[] = [];

const valid = await verifySolution(tokens, badSolutions, {
  onWarning: (w) => warnings.push(w.reason),
});

// warnings will contain 'invalid-solution' etc.
```

Das Feld `reason` im Warnungsobjekt kann einen der folgenden Werte haben:

| Reason | Wann er ausgelöst wird |
| --- | --- |
| `invalid-token` | Das JWT ist fehlerhaft, manipuliert oder verwendet ein unbekanntes Secret |
| `expired-token` | Die TTL der Challenge ist abgelaufen, bevor die Lösung eingereicht wurde |
| `invalid-solution` | Die eingereichte Nonce erzeugt keinen Hash mit den erforderlichen führenden Nullen |
| `replay-detected` | Dieselbe Token-JTI wurde bereits vom Replay-Store verbraucht |
| `configuration-error` | Eine erforderliche Option (z. B. `replayStore`) fehlt oder ist falsch konfiguriert |

<Warning>
  `solveChallenge` ist synchron und CPU-intensiv. Verwenden Sie es nicht in Produktions-Request-Handlern — es ist nur für Tests und Tooling konzipiert.
</Warning>
