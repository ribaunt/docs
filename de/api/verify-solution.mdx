`verifySolution()` wird serverseitig in deinem Verify-Endpunkt aufgerufen, um zu prüfen, ob der Browser alle Challenge-Tokens korrekt gelöst hat. Es validiert die JWT-Signatur, den Token-Ablauf, den SHA-256-Hash-Beweis, Replay-Status und optionalen Kontext.

## Import

```ts
import { verifySolution } from 'ribaunt';
```

## Signatur

```ts
function verifySolution(
  token: ChallengeToken | ChallengeToken[],
  nonce: number | string | Array<number | string> | ChallengeSolution | ChallengeSolution[],
  options?: VerifySolutionOptions
): Promise<VerifySolutionResult>
```

## Parameter

<ParamField path="token" type="ChallengeToken | ChallengeToken[]" required>
  Das/die ursprüngliche(n) JWT-Token, das/die von `createChallenge()` zurückgegeben wurde(n). Übergib dieselben Tokens, die dein Challenge-Endpunkt ausgegeben hat.
</ParamField>

<ParamField path="nonce" type="number | string | Array<number | string> | ChallengeSolution | ChallengeSolution[]" required>
  Die vom Browser eingereichte(n) Lösung(en). Kann sein:

  - Eine einzelne Nonce-Zeichenkette
  - Ein Array von Nonce-Zeichenketten
  - Ein `ChallengeSolution`-Objekt `{ nonce: string; hash: string }`
  - Ein Array von `ChallengeSolution`-Objekten (was das Widget als `solutions` sendet)
</ParamField>

<ParamField path="options" type="VerifySolutionOptions">
  Optionales Konfigurationsobjekt. Siehe die Optionstabelle unten.
</ParamField>

## Optionen

| Option | Typ | Standard | Beschreibung |
| --- | --- | --- | --- |
| `replayPrevention` | `'local' \| 'remote' \| 'disabled'` | `'local'` | Steuert, wie die Wiederverwendung von Tokens verhindert wird |
| `replayStore` | `ReplayStore` | `undefined` | Erforderlich, wenn `replayPrevention` auf `'remote'` gesetzt ist |
| `context` | `string` | `undefined` | Erfordert denselben Kontextwert, der beim Ausstellen der Challenge verwendet wurde |
| `debug` | `boolean` | auto (in dev `true`) | Protokolliert Verifizierungs-Warnungen in der Konsole |
| `onWarning` | `(warning: VerifyWarning) => void` | `undefined` | Callback für strukturierte Warn-Ereignisse |

## Rückgabewert

`Promise<VerifySolutionResult>`. Die Funktion gibt entweder

```ts
{ valid: true }
```

oder

```ts
{ valid: false, reason, message }
```

zurück.

## Beispiele

Grundlegende Verwendung:

```ts
const { tokens, solutions } = req.body;
const result = await verifySolution(tokens, solutions);

if (!result.valid) {
  return res.status(400).json({ success: false, error: result.reason });
}
```

Mit strukturierter Warn-Telemetrie:

```ts
const result = await verifySolution(tokens, solutions, {
  context: 'signup',
  onWarning: (warning) => {
    // warning.reason: 'invalid-token' | 'expired-token' | 'invalid-solution' | 'context-mismatch' | 'replay-detected' | 'configuration-error'
    console.log('captcha warning', warning.reason, warning.message);
  },
});
```

Mit Remote-Replay-Store:

```ts
const result = await verifySolution(tokens, solutions, {
  replayPrevention: 'remote',
  replayStore: {
    consume: async (jti, expiresAt) => {
      // Atomic set-if-not-exists with TTL (e.g. Redis SET NX EXAT)
      return await redis.set(jti, '1', 'NX', 'EXAT', expiresAt) !== null;
    },
  },
});
```

## Warn-Gründe

| Grund | Beschreibung |
| --- | --- |
| `invalid-token` | JWT-Signatur ist ungültig oder fehlerhaft |
| `expired-token` | Die Challenge-TTL ist abgelaufen |
| `invalid-solution` | Der Nonce erzeugt keinen gültigen Hash |
| `context-mismatch` | Der übergebene Kontext stimmt nicht mit dem Token-Kontext überein |
| `replay-detected` | Token wurde bereits verbraucht |
| `configuration-error` | `replayPrevention` ist `'remote'`, aber kein `replayStore` wurde bereitgestellt |
